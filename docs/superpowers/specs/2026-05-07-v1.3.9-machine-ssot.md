# v1.3.9 Spec 1 — 机器级 SSOT 完整化 (machine-ssot)

> **Status:** brainstormed 2026-05-07; awaiting user review before plan generation.
> **Scope of this spec:** A.1 + A.2 + A.3 + A.4 from `ROADMAP.md::v1.3.9`. These four items share a single `~/.feishu-user-plugin/` file layout + a single owner-based control loop and must be designed together.
> **Out of scope:** B.1 / B.4 cookie protobuf 抓包（Spec 2），D / F.1 小项（Spec 3），C md→wiki / B.5 search_messages / E config.js 拆分 / G OpenClaw（v1.3.10）。

## 1. Background & Goals

v1.3.7 引入了 `~/.feishu-user-plugin/credentials.json` 作为 cookie / UAT / app credentials / profileHints 的机器级 SSOT。v1.3.8 加了 in-memory WS event buffer，但**每个 MCP 进程一份独立 WS + 独立 buffer**，导致：

1. 多 harness 重复事件（Claude Code MCP 和 Codex MCP 都连了一个 WS，每条飞书 push 写两份）
2. 事件分散在 N 个进程内存里，没法跨 harness 共享或在重启间留存
3. `switch_profile` 在一个 harness 里调完，其它 harness 不感知（active 的迁移没 plumb 到位）
4. setup CLI 非交互模式（`--app-id X --app-secret Y`）跳过了 pointer-only 询问，每个 harness env 仍写一份完整 LARK_*——SSOT 还有漏点
5. WS 只订阅 `im.message.receive_v1` 一种事件，扩展受限

**v1.3.9 目标：** 把 WS / active profile / setup / event subscriptions 全部收敛到机器级 SSOT，让 "安装即机器级" 是默认路径，让多 harness 看到一致的实时事件流。

**Non-goals：**
- 不改变 v1.3.8 已有的 `get_new_events` 工具签名（filter 字段保持兼容；新增 `profile` filter 是可选项）
- 不引入新的进程间通信机制（IPC / socket）—— 全部通过文件 + advisory locks
- 不支持 broadcast 语义（同一事件被多个 consumer 各自看到一遍）—— 单一 cursor，**每条事件只被消费一次**

## 2. File Layout（v1.3.9 之后 `~/.feishu-user-plugin/`）

```
~/.feishu-user-plugin/
├── credentials.json              # SSOT（v1.3.7+；本版扩 schema）
├── ws-owner.lock                 # NEW：owner 单进程锁；mtime = heartbeat
├── events.jsonl                  # NEW：append-only event log
├── events.cursor.json            # NEW：单一全局 drain 游标
├── events.cursor.json.lock       # NEW：proper-lockfile 管理的 cursor 互斥锁
└── events.jsonl.dropped-<ts>     # 可选：force-rotate 时保留的旧 log（证据）
```

`~/.claude/feishu-uat-refresh.lock`（v1.3.5+）保留不变。

### 2.1 credentials.json schema 扩展（A.4 引入）

```json
{
  "version": 1,
  "active": "default",
  "profiles": {
    "default": {
      "LARK_APP_ID": "...",
      "LARK_APP_SECRET": "...",
      "LARK_COOKIE": "...",
      "LARK_USER_ACCESS_TOKEN": "...",
      "LARK_USER_REFRESH_TOKEN": "...",
      "events": ["im.message.receive_v1"]      // NEW，可选
    }
  },
  "profileHints": { ... }                      // v1.3.8
}
```

`events` 字段缺省时默认 `["im.message.receive_v1"]`，保持 v1.3.8 行为零回归。

### 2.2 ws-owner.lock 内容

```json
{ "version": 1, "pid": 12345, "start_time": 1715000000 }
```

写入方式：`fs.openSync(path, 'wx')`（即 `O_WRONLY|O_CREAT|O_EXCL`）一次写完成。
心跳：`fs.utimesSync(path, now, now)` 每 15 秒一次。

### 2.3 events.jsonl 单行格式

每行一个事件，行尾必须 `\n`：

```jsonl
{"event_id":"e_abc","ts":1715000000123,"profile":"default","payload":{...}}
```

字段：
- `event_id` — 飞书 SDK 给的，未必单调；用于 consumer 端 dedup（虽然单 cursor 已避免重复，但留作 debug）
- `ts` — 写入毫秒时间戳
- `profile` — **写入时 owner 的 wsProfile**（不是 active；区别见 §4.4）
- `payload` — SDK 原始 event 对象 JSON 化

**System events 约定**：插件自身合成的事件用 `event_id` 前缀 `_`（下划线），`profile = "_system"`。当前已定义：

- `_rotated` — force-rotate 时写入新 events.jsonl 的第一行；payload `{warning, prev_size, dropped_file}`

consumer 可以按 `profile != "_system"` 过滤，或者按 event_id 前缀 `_` 跳过。`get_new_events` 默认 profile 过滤会让 `_system` 事件**不返回给单 profile 用户**——如果要看 system events 必须显式 `profile: "*"`。

### 2.4 events.cursor.json

```json
{ "version": 1, "file": "events.jsonl", "offset": 4823 }
```

`offset` ∈ [0, events.jsonl 当前 size]。原子写：tmp + rename。

## 3. A.1 WebSocket 机器级（详细设计）

### 3.1 高层架构

- **单 owner / 机器**：哪个 MCP 进程拿到 `ws-owner.lock`，由它启 `WSClient` 把 SDK 的 `EventDispatcher` 回调写到 `events.jsonl`
- **所有进程读**（包括 owner）通过同一个 cursor 读 `events.jsonl`——`get_new_events` 走的就是这条路径
- **failover**：owner 死 → 心跳 mtime 60s 内不更新 → 任何进程下次 30s 心跳检测到 stale，原子抢锁，成功者成为新 owner

### 3.2 Owner 健康 / 接管协议

**owner 心跳计时器（每 15s）：**
1. `fs.utimesSync(ws-owner.lock, now, now)` 更新 mtime
2. `fs.statSync(credentials.json)`：mtime 变 → 触发 §4 active profile / §5 events list 检查
3. **rotation 检查**（§3.4）

**非 owner 心跳计时器（每 30s）：**
1. `fs.statSync(ws-owner.lock)`：`now - mtime > 60s` → 视为 stale → 尝试接管
2. **接管原子性**：`fs.renameSync(ws-owner.lock, ws-owner.lock.stale-<pid>)` → `fs.openSync(ws-owner.lock, 'wx')`。`rename` 失败 = 别人比我快，回退 non-owner；`openSync` ENOENT 也类似。竞争天然互斥不需要额外锁
3. 成功接管 → repair pass（§3.5）+ 启动 WS

**新 owner 启动顺序：**
1. 读 credentials.json 当前 active profile
2. 用该 profile 的 APP_ID/SECRET 构 WSClient
3. 用 profile 的 `events` 列表构 EventDispatcher（缺省 `["im.message.receive_v1"]`）
4. WS 握手成功 → 设 `wsProfile = active`，开始把事件写 events.jsonl

### 3.3 cursor 表示与 drain 协议

**cursor 表示：** `{file, offset}`，理由见 brainstorm 决策（A 选项）——byte offset 简单，rotation 用 file 字段处理。

**Drain 流程**（任何进程的 `get_new_events`）：

```
1. proper-lockfile.lock(events.cursor.json)  // 30s stale
2. let cur = readJSON(events.cursor.json) 或 缺省 {file: "events.jsonl", offset: <events.jsonl size>}
3. sanity: cur.offset ≤ statSync(events.jsonl).size (不通过 → 走 R1 保守 reset)
4. 从 cur.offset 读 events.jsonl 到 EOF，按 \n 切行
5. 最后一行没 \n → 视为 partial，本批不消费这一行（advance offset 只到上一行末）
6. 解析每行 JSON，apply filter（profile / event_type / chat_id / since_seconds）
7. 如果 peek=false: writeAtomic(events.cursor.json, {file, offset: 新值})
8. proper-lockfile.unlock
9. 返回 filtered events
```

**peek=true**：跳过第 7 步——offset 不动，下次仍能看到这批事件。

### 3.4 Events.jsonl rotation

**触发**（owner 心跳里）：
- `events.jsonl size > 10 MB` AND
- `events.cursor.json::offset >= events.jsonl size - 4 KB`（消费者已基本跟上；4 KB 阈值容忍最后几行未消费）

**defer-rotate 路径**（消费者已跟上）：
1. 取 cursor lock
2. `fs.renameSync(events.jsonl, events.jsonl.dropped-<ts>)`（保留旧 log 一段时间）
3. 创建新 events.jsonl 文件（owner 的 append 流自动重新打开）
4. `writeAtomic(events.cursor.json, {file: "events.jsonl", offset: 0})`
5. 释放 cursor lock

**force-rotate 路径**（消费者没跟上 + log > 20 MB）：
- 同上，但写一行 stderr warning + 写一条标记事件（`{"event_id":"_rotated", "ts":<>, "profile":"_system", "payload":{"warning":"force_rotated_log","prev_size":<>}}`）
- 一段时间后清理 `events.jsonl.dropped-<ts>`：保留 7 天，owner 心跳里加一个老旧文件清扫（每小时一次足够）

**多代旧 log（events.jsonl.1 / .2 / ...）不实施**——defer + force 两层够用，不引入跨文件 cursor 跳转的复杂度

### 3.5 Repair pass（新 owner 接管时跑一次）

```
1. let stat = fs.statSync(events.jsonl)
2. 从末尾倒读 1 KB
3. 找最后一个 \n
4. 如果末尾不是 \n → 截断到最后一个 \n 位置（fs.truncateSync）
```

确保 append-only invariant：每行都以 `\n` 结尾，新 owner append 不会把旧 partial 拼到自己的事件里。

### 3.6 Defenses（R1-R7）

| ID | 风险 | 防御 |
|---|---|---|
| R1 | cursor.json 损坏 / 解析失败 / sanity 失败 | 保守 reset 到 `{file: events.jsonl, offset: <current size>}`（**绝不重置 0**），stderr warning |
| R2 | rotation 跨文件 cursor | defer-rotate 优先 + force-rotate 兜底；不引入多代旧 log |
| R3 | 并发 drain | proper-lockfile on events.cursor.json，30s stale |
| R4 | owner 崩溃留 partial line | 写侧 `appendFileSync(line + '\n')` 单 syscall；读侧 partial 不消费；新 owner repair pass |
| R5 | events.jsonl 被 rm | read 时 ENOENT → 视为空，cursor reset 到新空文件 offset 0 |
| R6 | disk full | owner 写失败 → 落回 v1.3.8 的 in-memory buffer（cap 1000）作为 disk-full fallback；3 次重试 × 指数退避失败后丢最老 + stderr。**v1.3.8 buffer 不再被 get_new_events 直接读**，仅作 owner-side disk-full 通道 |
| R7 | cursor sanity 通过但语义错 | 原子写 + tmp+rename 已把概率降到 fs 故障级别，不实施 checksum |

### 3.7 manage_ws_status 工具

**Schema：**

```js
{
  action: "info" | "reconnect" | "claim" | "rotate" | "reconfig",
  force?: boolean    // 仅 claim 用：force=true 时强夺活 owner 锁
}
```

**5 个 action 行为：**

| action | 谁能调 | 行为 | 副作用 |
|---|---|---|---|
| `info` | 任何进程 | 返回机器 / owner / WS / log / cursor / config 状态（schema §3.7.1） | 无 |
| `reconnect` | 仅 owner | 关 WS + 用当前配置重连 | WS ~5-10s 不可用 |
| `claim` | 任何进程 | lock 已 stale → 直接接管；活 owner + `force=true` → 强夺锁 + 旧 owner 下次心跳发现锁不对自降 non-owner | WS 切换 ~5-10s |
| `rotate` | 仅 owner | 强制 events.jsonl rotation（即使没到 10 MB） | cursor 重置 0 |
| `reconfig` | 仅 owner | 立刻读 credentials.json 最新 active + events list，diff 后必要时切 WS（A.2 流程） | WS 切换 ~5-10s |

非 owner 调 owner-only action → 返回 `{error: "not_owner", owner_pid: <>}`。

#### 3.7.0 各 action 返回 schema

- `info` → §3.7.1 完整状态 dump
- `reconnect` → `{ok: true, ws_state: "switching"}`（异步，调用方需后续 info 看 state）
- `claim` → `{ok: true, became_owner: true, prev_owner_pid: <>}` 或 `{ok: false, reason: "lock_active_no_force"}`
- `rotate` → `{ok: true, prev_size: <>, dropped_file: "events.jsonl.dropped-<ts>"}`
- `reconfig` → `{ok: true, ws_state: "switching", prev_subscriptions: [...], next_subscriptions: [...]}` 或 `{ok: true, no_change: true}` 当 diff 为空

#### 3.7.1 `info` 返回 schema

```json
{
  "this_process": {
    "is_owner": true,
    "pid": 12345,
    "start_time": 1715000000
  },
  "owner": {
    "pid": 12345,
    "start_time": 1715000000,
    "last_heartbeat_age_seconds": 7,
    "alive": true
  },
  "ws": {
    "state": "connected",                          // connected | switching | disconnected
    "wsProfile": "default",                        // currently authenticated profile name
    "subscribed_events": ["im.message.receive_v1"],// what's actually registered with the SDK now
    "last_reconnect_at": 1715000000,
    "reconnect_attempts": 0
  },
  "log": {
    "size_bytes": 1234567,
    "last_write_age_seconds": 3,
    "cursor_offset": 1230000,
    "events_pending_bytes": 4567,
    "rotation_count_today": 0
  },
  "config": {
    "active_profile": "default",
    "configured_events": ["im.message.receive_v1"] // 来自 credentials.json::profiles[active].events
  }
}
```

`ws.subscribed_events` 是**当下 WS 实际订阅**的，`config.configured_events` 是**配置文件里写的**——切换中间会短暂不一致（reconfig 在路上）。两者持续不一致 = 配置里有非法 event string 被 SDK 拒绝（参考 §6.2 容错）。

非 owner 调 info：`this_process.is_owner=false`，`ws` 段省略（不知道）。

## 4. A.2 Active profile 跨进程同步

### 4.1 触发机制

**dispatcher 入口同步检查**（每个 MCP tool call 之前）：

```
1. fs.statSync(credentials.json)
2. 若 mtime 与 in-memory baseline 不同 → readJSON
3. 若 active 与 in-memory currentProfile 不同 → setActiveProfile(new)
4. 进入 tool handler
```

成本：~10μs / call。**纯 SSOT，env `FEISHU_PLUGIN_PROFILE` 失效**——env 仅作首次启动 bootstrap（如 credentials.json 没 active 字段）。

### 4.2 Cache invalidation 范围

setActiveProfile 清下面**所有** profile 关联状态：

- `userClient`（cookie 会话）
- `officialClient`（APP_ID/SECRET 实例）
- UAT / OAuth client
- profileHints 缓存（自动切 profile 用的 resourceKey → profile 表）
- resourceKey 解析缓存（chatId / wikiNode 等——为防同 ID 在不同 bot 上下文有不同权限）
- WS owner 状态：见 §4.4

### 4.3 Per-event profile tag

events.jsonl 每行已含 `profile` 字段（§2.3）。`get_new_events` 过滤增加 `profile` 维度：

```js
get_new_events({
  profile?: string,        // 缺省 = currentActive；"*" / "any" = 不过滤
  event_type?: string,
  event_types?: string[],
  chat_id?: string,
  since_seconds?: number,
  peek?: boolean
})
```

**默认行为变化**：未指定 `profile` 时只返回当前 active 的事件。其它 profile 事件**消费但不返回**（cursor 推进，不重复见到）。

**向后兼容**：单 profile 用户所有事件 `profile="default"`，currentActive 永远 `"default"`，过滤无副作用。

### 4.4 WS profile 切换流程（owner 心跳触发）

owner 维护两个独立字段：
- `wsProfile`：当前 WS 鉴权用的 profile 名（如 `"default"` / `"alt"`）
- `wsState`：连接生命周期，`"connected" | "switching" | "disconnected"`

owner 心跳 tick 检测到 active 变（§3.2 第 2 步）→：

```
1. wsState = "switching"           // 期间收到的事件被丢弃（不知道归哪个 profile）
2. WS.close() (graceful, 1-2s)
3. reload officialClient with new active's APP_ID/SECRET
4. 用新 profile 的 events 列表构 EventDispatcher
5. WS.connect()                    // 5-8s
6. wsProfile = newActive
7. wsState = "connected"
```

总停机 ~5-10s。在此期间收到的飞书 push 不写 events.jsonl，consumer 看到这段时间空白。**用户可接受**（profile 切换是低频显式操作）。

`wsProfile` 在切换中**保持旧 profile 名**直到第 6 步——给诊断用（"现在还在 default 在转移"），不参与事件 tag（事件在 wsState != "connected" 时直接丢，根本不写）。

### 4.5 异常分支

- **owner idle**（没 tool call）：心跳计时器独立于 dispatcher，仍 tick → 15s 内必检测 active 变化
- **owner 死了**：60s stale → 新 owner 接管 → 接管时直接读当下 active 起 WS（不需要切换流程）
- **快速 A→B→C**：心跳一次性看到 active=C，跳过 B 直接到 C；events.jsonl 里写入时刻的 wsProfile 反映实际状态
- **多进程同时 setActiveProfile**：credentials.json 的 atomic write 排他；后写赢

## 5. A.3 setup CLI 自动机器级

### 5.1 决策矩阵

| credentials.json | env LARK_* | 用户给 `--app-id` | setup 行为 |
|---|---|---|---|
| ❌ 不存在 | 无 | 给了 | **全新安装**：写 credentials.json default profile（含 --app-id/--app-secret/--cookie 给的所有凭证）；harness env 只写 `FEISHU_PLUGIN_PROFILE=default` |
| ❌ 不存在 | 有（v1.3.8 老用户）| 给/不给 | **自动迁移**：env 里 LARK_* 移入 credentials.json default profile；如果有 --app-id 合并覆盖；harness env 改 pointer-only。等价于 setup 内部 `migrate --confirm` |
| ✅ 存在 | 有/无 | **不给** | **保持 SSOT**：不动凭证；harness env 残留的 LARK_* 清掉，写 `FEISHU_PLUGIN_PROFILE=<active>` |
| ✅ 存在 | 有/无 | **给了** | **更新凭证**（§5.2 的 B 策略）|

### 5.2 已存在 credentials.json + 用户给 --app-id 的处理

**默认拒绝 + 显式标志**：

```
$ npx feishu-user-plugin setup --app-id NEW --app-secret SECRET
error: 'default' profile already exists; pass --force to overwrite, or --profile <name> to create a new profile
```

- `--force` → 覆盖 default profile 凭证 + warning 提示
- `--profile alt` → 创建新 profile，default 保留；提供"注册多 profile"的 first-class 入口

**`LARK_PROFILES_JSON` env 向后兼容**：v1.3.8 的 env-driven 多 profile 注册路径继续工作，没有 deprecation。setup --profile 是新增的 first-class 入口（写入 credentials.json），适合"我装好后才想加 profile"的场景；env 仍适合 "harness 启动前从外部注入"的场景。**两条路径并存，不冲突**。

### 5.3 --client both 一致性

`--client both` 写两份 pointer-only env（`~/.claude.json` + `~/.codex/config.toml`），共享同一份 `~/.feishu-user-plugin/credentials.json`。两个 harness 自动 SSOT。

### 5.4 migrate CLI 不变

migrate 命令保留作为独立 CLI（v1.3.7+ 行为不变）。setup 在矩阵第 2 行内部调用 `migrate --confirm`——只是触发路径变了，行为相同。

## 6. A.4 Event type expansion

### 6.1 Schema 与默认

见 §2.1。`profiles[*].events` 字段，可选，默认 `["im.message.receive_v1"]`。

### 6.2 不预先 hardcode 事件类型清单

接受任意 string。owner 启 WS 时：
- 遍历 events 列表逐个 `EventDispatcher.register(name, handler)`
- 单个 register 失败 → stderr warning + 跳过该 string
- 全部失败 → 兜底 register `im.message.receive_v1`，stderr 警告 `"all configured events rejected, fallback to default"`

CLAUDE.md / README 给参考清单（不强制白名单）：
- `im.message.receive_v1`（默认）
- `approval.instance.created_v4` / `approval.instance.cc_v4`
- `calendar.calendar.event.changed_v4`
- 文档评论事件（具体 string 待 SDK 文档查；不在 v1.3.9 spec 范围内验证）

### 6.3 修改路径（3 个口子）

1. **手动编辑 credentials.json**（最简单）
2. **`manage_ws_status(action=reconfig)`** —— 编辑后无需重启 MCP，主动让 owner 重读 + 重订阅
3. **环境变量 `FEISHU_PLUGIN_EXTRA_EVENTS=approval.instance.created_v4,calendar.event.changed_v4`** —— **仅 first-time bootstrap**：credentials.json 无 events 字段时初始化用一次。已有 events 字段时 env **完全失效**（保持纯 SSOT）

不加新 CLI 命令（`events add` / `list` / `remove`）——v1.3.10 看用户反馈再决定。

### 6.4 自动重订阅与 A.2 心跳整合

owner 心跳 stat credentials.json mtime 变 → 重读 active profile：
- active 变 → §4.4 切换流程
- active 没变但 events 列表变 → 同样走切换流程（停机 ~5-10s 重连 + 用新清单 register）

逻辑统一：**改 events 列表 / 切 active / 改 APP_ID 都走同一条 WS 重配响应路径**。

未来优化：如果 SDK 支持增量 add/remove subscription（不重连 WS），fast path 可降到 ~1s 不停机。**不在 v1.3.9 范围**——先全量重连保证正确性。

## 8. Open Questions / Risks

### 8.1 SDK 不支持增量订阅
当前 `@larksuiteoapi/node-sdk` 的 `EventDispatcher` API 似乎是 register-then-connect 一次性的。**v1.3.9 不假设增量支持**，每次 events 列表变 / active 变都全量重连。stop-the-world ~5-10s 是已知代价，纳入文档。

### 8.2 飞书 SDK WSClient 与 events.cursor.json 的解耦
SDK 只负责事件入站（owner 内的 EventDispatcher 回调）。events.jsonl / cursor / lock 全部由本插件实现，不依赖 SDK 任何持久化能力。这一边界要在 implementation 中明确。

### 8.3 Force-rotate 数据丢失
force-rotate（log > 20 MB 仍未消费）会把旧 log 改名为 `events.jsonl.dropped-<ts>`，cursor 重置 0。理论上 consumer 还能去读 dropped 文件——但**当前 spec 不暴露读取 dropped log 的工具**，consumer 没跟上就是真丢了。如果 v1.3.9 用户反馈这是问题，v1.3.10 加一个 `manage_ws_status(action=read-dropped)`。

### 8.4 PID 复用
heartbeat 用 mtime 而非 PID 探活，所以 PID 复用不会误判。但 manage_ws_status info 里的 owner.pid 字段仅诊断用，不参与失效判定。

### 8.5 多机器场景
本 spec 假设所有 harness 在**同一台机器**共享 `~/.feishu-user-plugin/`。NFS / 网络盘 / 远程 home 的场景**未测试**。advisory locks 在 NFS 上行为不可靠。文档里建议 `LARK_HOME` 必须是本地盘（或类似明示）。

## 9. Migration Notes（v1.3.8 → v1.3.9）

升级时的用户感知：

1. 第一次启动 v1.3.9 MCP：检测到 `~/.feishu-user-plugin/` 没 ws-owner.lock / events.jsonl → owner 抢锁创建文件，正常运转。**v1.3.8 的 in-memory buffer 不再被读**，但 disk-full fallback 仍保留为兜底通道。

2. 第一次跑 `npx feishu-user-plugin setup`：根据 §5.1 矩阵自动收敛——v1.3.8 老用户走自动迁移分支。

3. 第一次调 `get_new_events`：cursor 文件不存在 → 用 `{file: events.jsonl, offset: <current size>}`（保守 reset 语义）。**v1.3.9 启动前积压的 v1.3.8 内存事件全部丢弃**——这是有意的，避免 v1.3.8 buffer 跨升级保活的复杂性。

4. CLAUDE.md 的 "Multiple MCP processes get duplicate events" 限制需要删除。

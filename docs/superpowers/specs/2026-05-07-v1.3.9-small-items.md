# Part A — D: `read_doc_markdown`

## 2. Goals

- 给 RAG / digest / 概要生成类的 agent 用例**省 60% token**（实测 90 / 216 ≈ 41% 大小，等价节省 ~58%）
- `get_doc_blocks` 仍保留，结构化遍历场景继续用
- 接受与 `read_doc` / `get_doc_blocks` 完全相同的 `document_id` 输入形式（native token / wiki node / Feishu URL）

## 3. Tool Signature

```js
read_doc_markdown({
  document_id: string,             // native docx token / wiki node / Feishu URL
  // No format options for v1.3.9 — keep minimal
})
```

返回：纯 markdown 字符串。

**为什么不加选项**（如 `include_metadata`、`heading_offset`、`max_length` 等）：
- v1.3.9 范围内是 token 优化工具，最小可用就够
- 真有用例再 v1.3.10 加，YAGNI 原则
- `read_doc` 也是返回原始内容，没花哨参数；保持 API 一致性

## 4. Implementation Strategy

### 4.1 库选型：`feishu-docx@0.7.0` 作主路径

- 公开 npm 包（MIT，一年前更新，相对成熟）
- 维护者 longbridgeapp/feishu-pages，有人在用
- 8 deps（含 jsdom + marked），~525 KB unpacked
- 转换 docx blocks → Markdown / HTML

主路径调用：

```js
const { transform } = require('feishu-docx');

async function read_doc_markdown(args, ctx) {
  const docId = await ctx.resolveDocId(args.document_id);
  const blocks = await ctx.getOfficialClient().getDocBlocks(docId);
  const md = transform(blocks);  // API 名待 implementation 时确认
  return text(md);
}
```

### 4.2 块类型覆盖与 fallback 策略

`feishu-docx` 不一定覆盖所有飞书 block 类型（飞书过去一年加了 callout / quote-container 等新 block）。

**Fallback：**
- 实施前先用 1 个真实文档（含尽可能多类型）跑一次 `feishu-docx::transform`，看哪些 block 被 silently 丢弃
- 缺失的类型按以下策略处理：
  - **简单可补**（heading 7-9 / divider / quote / todo / 新版 callout）→ 直接在 plugin 内写一小层 post-processor，把缺漏的 block 类型转 markdown 等价物
  - **embed 类**（bitable / sheet / mindnote / 子文档）→ markdown 输出占位 `> [Bitable: <name>](feishu://base/<token>)` 这种链接形式，agent 想读取可二次调用 `read_doc` / `manage_bitable_record`
  - **不重要的稀有 block**（特殊媒体 / 实验性）→ 跳过，markdown 里加 HTML 注释 `<!-- block type N skipped -->`

**不预先列死哪些要补**——implementation 时跑一次真实文档对比，按真实缺失情况补。

### 4.3 Embedded resources（图片 / 文件）

文档里的图片 / 文件块怎么变 markdown：

- **图片**：`![alt](feishu://image_token/IMG_TOKEN)` —— **占位 URL scheme**，agent 想拿真图调 `download_doc_image(IMG_TOKEN, doc_token=DOCID)`（已存在的工具）
- **文件**：`[FILE_NAME](feishu://file_token/FILE_TOKEN)` —— 同样占位
- **不自动下载**：read_doc_markdown 是只读 + 省 token 工具，下载是高成本动作不该 implicit

`feishu://` URL scheme 是我们自己定的伪 URL，便于 agent 一眼看出"这是飞书资源 token"——不是 web 上能解析的真 URL。

如果 `feishu-docx` 默认输出别的 URL 形式，post-processor 把它统一改成 `feishu://...`。

### 4.4 文件位置

- `src/tools/docs.js` 已存在，加 `schemas` 项 + `handlers.read_doc_markdown`
- `src/clients/official/docs.js` 不需要新方法（复用 getDocBlocks）
- 新依赖：`package.json` 加 `feishu-docx@^0.7.0`

### 4.5 Token 节省 verification

实施时用真实文档测：

```
docId: 一份团队真实大文档（≥100 blocks）
size_a = JSON.stringify(getDocBlocks(docId)).length
size_b = read_doc_markdown(docId).length
ratio = size_b / size_a
```

ROADMAP 声称 ~41% (90/216)，目标实测在 30-50% 之间。低于 30% 说明实现有 bug；高于 50% 说明 markdown 还不够紧凑（比如保留了过多元数据），需要 post-processor 调整。

## 5. Test Plan (D)

### 5.1 单元

### 5.2 集成

  - 输出 markdown 渲染没崩
  - 包含的 block 类型都有合理输出
  - token 节省 ratio 在 30-50%
- 不进 CI（需要真实 cookie 和文档权限）

# Part B — F.1: `switch_profile` 多 profile e2e

## 6. Goals

- 验证 `switch_profile` MCP 工具调用后，**当前进程内的 cached `userClient` / `officialClient` / UAT client 真的被 invalidate**，下一次 tool call 用新 profile 的凭证
- 验证 `~/.feishu-user-plugin/credentials.json::active` 字段被原子写入新值
- 验证 profileHints / resourceKey 缓存被清

**不验证**：
- 跨进程同步（A.2 的范畴，Spec 1 §7.2 的 multi-process 测试覆盖）
- 真实 Feishu API 调用成功（用 dummy 凭证，验证仅看 **client 实例的内部凭证字段**——不发真实网络请求，CI 能跑）

## 7. Test Mechanism

### 7.1 测试隔离

- 测试启动前：备份 `~/.feishu-user-plugin/credentials.json` 到 `tmp/cred-backup-<ts>.json`
- 测试结束后（success 或 fail）：原文复原
- 用 try/finally 确保即使 assertion 失败也复原
- **不修改用户的真实 default profile 凭证**——只追加一个 dummy `alt` profile 到 profiles 字典

### 7.2 dummy alt profile

```json
{
  "profiles": {
    "default": { /* 用户原 */ },
    "alt": {
      "LARK_APP_ID": "cli_test_alt_xxxxxxxx",
      "LARK_APP_SECRET": "test_secret_alt",
      "LARK_COOKIE": "session=fake_alt_cookie",
      "LARK_USER_ACCESS_TOKEN": "u-fake-alt-uat",
      "LARK_USER_REFRESH_TOKEN": "fake-alt-refresh"
    }
  }
}
```

dummy 凭证的 APP_ID 含有可识别字符串 `test_alt`，便于断言"这次请求带的是不是 alt 凭证"。

### 7.3 断言点

测试**不开 stdio MCP 协议**——直接 `require('./server.js')` 拿到 ctx 工厂，构造 ctx，**直接 call 工具 handler 函数**（绕过 MCP transport 层）。这是 v1.3.7 重构后的好处：handler 是纯函数 `(args, ctx) → result`，可单独调用。

跑下面序列：

```
1. ctx.getOfficialClient() → client_default; 记下其内部 APP_ID 字段
2. tool('switch_profile', {profile: 'alt'}) → expect ok
3. credentials.json 重读 → expect active === 'alt'
4. ctx.getOfficialClient() → client_alt; expect 与 client_default 不是同一个对象引用
5. client_alt 的内部 APP_ID 含 "test_alt"
6. ctx.profileHintsCache → expect empty / reset
7. ctx 里其它 profile-bound 缓存（resourceKey 等）→ expect cleared
8. tool('switch_profile', {profile: 'default'}) → 切回
9. credentials.json::active === 'default'
10. ctx.getOfficialClient() → 与 client_alt 不同；与 client_default 也不一定相同（重建后是新实例）
```

### 7.4 测试集成

- 新文件：`src/test-switch-profile.js`
- 加进 `src/test-all.js`：跟现有 test 风格一致（plain node assertions，不引 jest）
- 顺序：放在已有 test 之后；如果失败会让 `npm test` 整体红
- 必要的"内部状态访问"：通过 ctx 暴露的 getter（如 `ctx.getOfficialClient()._creds` 测试用）；如果没暴露，加 test-only export 而不是改 production code

### 7.5 cleanup 保证

如果测试中途 process exit（panic / 别的 bug），credentials.json 不能留在脏状态：
- backup 文件路径基于 timestamp，不会冲突
- 测试入口顺手扫 `~/.feishu-user-plugin/cred-backup-*.json` 老文件，发现就提醒"前次测试可能没复原，手动检查"

## 8. Risks (F.1)

### 8.1 测试与并行 MCP 进程冲突

测试运行时 credentials.json 被改 → 真实运行的 Claude Code MCP 可能透过 A.2 的 dispatcher stat 检测到 → setActiveProfile 切到 alt → 用户工具调用瞬间报错。

**缓解**：
- README.md `npm test` 一节加红色警告："运行此测试会临时修改 ~/.feishu-user-plugin/credentials.json，建议先停止运行中的 Claude Code / Codex MCP（pkill -f feishu-user-plugin）"
- 测试启动前 print 一行 stderr 警告，让用户有 chance 取消
- v1.3.10 可考虑用环境变量 override credentials.json 路径（让测试用独立 fixture 文件），但**不在 v1.3.9 范围**

### 8.2 测试在 CI 跑

CI 没有 `~/.feishu-user-plugin/credentials.json`（没人 setup 过），测试启动 backup 阶段会发现没文件——这种 case 走"fresh init"路径：写一个 fresh credentials.json 当 backup 即可。测试结束删除整个文件。

CI 也不需要真实 LARK_APP_ID/SECRET——dummy 凭证只用于 client 实例化和断言比对，不需要联网。

### 8.3 ctx 内部状态暴露

如果 `ctx.profileHintsCache` 这种内部状态在 production code 里没 getter，要么：
- 加 production-safe getter（小代码改动，可接受）
- test-only export `ctx.__test_internalState`（侵入性更小）

implementation 时按 ctx 的 v1.3.8 现状决定。

## 10. Open Questions

### 10.1 feishu-docx 不维护 / 缺失关键 block 类型
如果实测发现 feishu-docx 缺得太多（比如 callout / 新版 todo），fallback 是 hand-roll 一个 minimal 转换器。500 行内能搞定。**不在 v1.3.9 spec 决定**——implementation 时根据真实文档评估。

### 10.2 Token 节省不达预期
如果实测节省比例 < 30%（比如 markdown 还有大量 metadata noise），可加 post-processor 删除：
- 折叠连续空行
- 删除空 paragraph
- 删除 inline style metadata

仍 < 30% → 重新审视方案，但**不强制 v1.3.9 ship**——可推到 v1.3.10。

### 10.3 测试 ctx 暴露的最小侵入路径
implementation 时根据 ctx 当前 schema 决定：production-safe getter vs test-only export。**不在 spec 锁定。**

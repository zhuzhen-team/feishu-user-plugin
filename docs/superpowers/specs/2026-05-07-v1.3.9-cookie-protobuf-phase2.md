# v1.3.9 Spec 2 — Cookie Protobuf Phase 2 (image + card)

> **Status:** brainstormed 2026-05-07; awaiting user review before plan generation.
> **Scope of this spec:** B.1 (`send_image_as_user`) + B.4 (`send_card_as_user` 真用户身份) from `ROADMAP.md::v1.3.9::B`. These two share the v1.3.8 capture/decode tooling and the same `_sendMsg` integration pattern.
> **Out of scope:** B.2 audio / B.3 sticker（v1.3.7→1.3.8→1.3.9 决定彻底删除，不做）, B.5 search_messages（推到 v1.3.10）, A 系列（Spec 1）, D / F.1（Spec 3）.

## 1. Background & Goals

### 1.1 当前状态（v1.3.8）

`src/clients/user.js::_sendMsg` 把消息走 cookie protobuf gateway。下面这些类型已能 work：

| 类型 | proto 字段 | 状态 |
|---|---|---|
| TEXT | `Content.text` / `Content.richText` | ✅ |
| FILE | `Content.fileKey` + `fileName` | ✅ |
| POST | `Content.richText`（rich text 元素图） | ✅ |
| IMAGE | `Content.imageKey` 单字段 | ❌ HTTP 400 |
| CARD | （proto 完全没字段）| ❌ 不存在 |

v1.3.8 已 ship 反向工程工具链（`scripts/decode-feishu-protobuf.js` 带 unknown field 报告 + `scripts/capture-feishu-protobuf.js` recipe + `docs/COOKIE-PROTOBUF-CAPTURES.md` 占位），但实际抓包工作推迟到本版。

### 1.2 目标

- 兑现 v1.3.7 → v1.3.8 推迟两次的承诺：用户身份（不是 bot）发图 + 发卡片
- 删除 user.js / 工具描述 / CLAUDE.md / README 里所有 "deferred to v1.3.9" 措辞
- B.1 / B.4 互相独立：捕获其一失败不阻挡另一个 ship

### 1.3 Non-goals

- 不实现 B.2 audio / B.3 sticker（v1.3.9 已确认删除）
- 不动 bot 路径：`send_message_as_bot(msg_type="image"/"interactive")` 仍是 fallback / 主路径并存
- 不破坏 `send_image_as_user(image_key=...)` 的现有签名——新增可选参数，不强制迁移
- 不实现"增量 wire format 协议升级机制"（飞书改协议时只能重抓重补，不在 v1.3.9 范围）

## 2. Methodology — 反向工程作为实施工作的一部分

**核心约束：实施会话必须包含真实抓包**——不是"假设 captures 存在"。

### 2.1 抓包协议（每个类型独立跑一遍）

由实施 agent 通过 Playwright MCP 驱动浏览器，在用户本机执行：

```
A. browser_navigate https://www.feishu.cn/messenger/
B. browser_run_code: 把 LARK_COOKIE 注入 context（如已登录则跳过）
C. browser_run_code: monkey-patch fetch — 拦 /im/gateway/ POST，把 body 存 window.__CAPTURED_GATEWAY_BODY__（base64）
D. browser_click: "我自己" 自聊
E. 触发发送：
   IMAGE → browser_file_upload 上传测试 PNG / JPEG / 大图（≥3 张，不同尺寸/MIME）
   CARD  → 见 §2.4 的特殊路径（web UI 不直接暴露卡片编辑器）
F. browser_evaluate: window.__CAPTURED_GATEWAY_BODY__ 拿 base64
G. 写到 /tmp/feishu-captures/<type>-<n>.b64
H. node scripts/decode-feishu-protobuf.js Content --b64 "$(cat ...)" — 看 known + unknown fields 报告
I. 把 unknown fields 加进 proto/lark.proto::Content（可能涉及新嵌套消息）
J. 重跑 H：直到 decoder 报告 "All fields known"
K. 重复 E-J 用其它测试样本，做交叉验证；任一样本出现新 unknown → 加进 proto，再回 J
```

抓包同时**保留 b64 文件作 regression fixture**：复制到 `tests/fixtures/captured/<type>-<n>.b64` + commit。后续编码逻辑必须能 round-trip 这些 fixture 字节相等（§5.2）。

### 2.2 至少抓 3 张 / 类型

- IMAGE：小 PNG（带 alpha）/ JPEG（无 alpha 不同 MIME）/ 大尺寸图（≥1024×1024）
  - 验证：尺寸 / MIME / 缩略图是否 conditional
- CARD：纯文本卡 / 含按钮卡 / 含 module 复杂卡
  - 验证：卡片复杂度对 wire format 字段集合是否有影响

单次抓包不够——曾在 IMAGE 上吃过亏（v1.3.7 错误地认为 imageKey 一字段就够，因为没看到飞书要求的元数据）。

### 2.3 Decode → patch 迭代

`decode-feishu-protobuf.js` 已能在 unknown wire 上报告 `field N (wire type W, L bytes): preview`。每次迭代：

1. 看每个 unknown field 的 wire type
2. wire 0 (varint) → 大概率 int / bool；wire 2 (length-delimited) → 字符串 / 嵌套消息 / bytes
3. 在 `proto/lark.proto::Content`（或新增子消息）加占位字段
4. 重跑 decoder，看新加字段是否能解析，preview 是否合理（字符串看起来像字符串、嵌套消息内部能进一步 decode）
5. 反复直到 "All fields known"

完成后所有未知字段都映射到了具体 proto 字段——**保守做法是先全量字段名匿名化（`unknownN`），跑通后再按语义重命名**（避免半懂不懂的猜测污染 schema）。

### 2.4 CARD 抓包的特殊路径

飞书 web UI 不直接暴露卡片编辑器。两条候选：

**i) 转发已有 bot 卡片**
- 先用 `send_message_as_bot(msg_type="interactive", payload=<test card>)` 发到自聊
- 在浏览器里点收到的卡片 → "转发" → 选自聊 → 这次发送会触发 `/im/gateway/` POST
- monkey-patch 抓到的就是用户身份转发卡片的 wire format
- 风险：转发可能用 `forward` 命令而非 `PutMessage`，cmd 不一定是 5。需观察实际 cmd 字段

**ii) 飞书 web 卡片 SDK 内嵌**
- 部分场景（机器人 webhook 测试页）会让用户从浏览器里手工发卡片。如果存在这种内部页面可绕过限制
- 需要探索

**iii) 兜底：cookie+UAT 协议层注入**
- 不通过 web UI，直接构造 `PutMessageRequest{type=14, content=...}`，用各种猜测填 content 字节，对比 200/400 收敛字段
- 慢、痛苦，但最后一步保险

抓包阶段先试 i，失败转 ii，最差 iii。**i 已在 docs/COOKIE-PROTOBUF-CAPTURES.md 标注为推荐路径。**

## 3. B.1 — `send_image_as_user` 设计

### 3.1 Proto 变化（具体字段待捕获结果）

预测（基于经验，待 §2 验证）：

```proto
// proto/lark.proto::Content 新增字段（字段号待 capture）

message Content {
  // 已有...
  optional string imageKey = 2;
  // 新增（推测）
  optional ImageMeta imageMeta = ??;       // 字段号待抓包
}

message ImageMeta {
  optional int32 width = 1;
  optional int32 height = 2;
  optional string mimeType = 3;
  optional int64 fileSize = 4;
  optional string thumbnailKey = 5;       // 可能是 thumbnail 是另一个 imageKey
  // 等待抓包结果
}
```

**不预先在 spec 里冻结字段名 / 字段号**——这是 capture 决定的。spec 提供的是"字段大致包含这些信息"的概念，实施时按真实抓到的填。

### 3.2 元数据来源策略（混合方案 D）

`send_image_as_user` 的元数据三条路径（由 fallback 顺序）：

```
1. 显式参数 width / height / mime（最高优先级）
   → 直接用，不查任何缓存
2. image_path 给本地文件
   → plugin 读文件用 image-size 库提取 width/height/mime/size
   → 如果没传 image_key，顺手 upload，拿 key
3. 缓存查询（image_key 唯一参数时）
   → 读 ~/.feishu-user-plugin/image-meta.json，按 image_key 查
   → hit：用缓存里的元数据
   → miss：报错 §3.4
```

### 3.3 元数据缓存：`~/.feishu-user-plugin/image-meta.json`

**Schema**

```json
{
  "version": 1,
  "entries": {
    "img_v3_xxx": {
      "width": 800,
      "height": 600,
      "mime": "image/png",
      "size": 12345,
      "thumbnail_key": "img_v3_yyy",
      "uploaded_at": 1715000000
    }
  }
}
```

**写入路径**：

- `upload_image` MCP 工具（Official API + 大概率插件已有的 cookie 上传路径）每次成功上传时，提取本地文件的元数据（`image-size` lib + `fs.statSync`）→ atomic merge 进 entries
- 不主动清理；自然 LRU 控制（cap 1000 entries，老的最先删）

**读取路径**：

- `send_image_as_user` 在缓存查询步骤（§3.2 step 3）读

**LRU**：超过 1000 entries → 按 `uploaded_at` 升序删最老的；阈值可配（不在 v1.3.9 暴露）

**dependency**：新增 `image-size` npm 包（纯 JS，~100 KB，0 native dep，覆盖 PNG/JPEG/GIF/WebP/BMP/SVG/TIFF/ICO 等）。如果项目避免增加 dep，fallback 路径 §3.5 记录手动实现 PNG/JPEG header 解析。

### 3.4 错误信息（缓存 miss + 没显式参数 + 没 image_path）

```
send_image_as_user: image metadata required for cookie wire format.
The image_key "img_v3_xxx" is not in the local upload-meta cache.
Pass one of:
  - image_path: <local file>     (will read meta from file, optionally upload + send)
  - width, height, mime          (explicit override; size optional)
  - re-upload via upload_image which now caches metadata
or fall back to send_message_as_bot(chat_id, msg_type="image", payload={image_key:"..."}).
```

清晰指向解决路径。

### 3.5 `image-size` dep 的 fallback

如果 user reviewer 反对引入 dep：
- 内联实现 PNG（IHDR chunk 偏移 16-23 = width/height）和 JPEG（SOF0 marker 扫描）的 header parser，~50 行代码
- MIME 从 magic bytes：PNG = `89 50 4E 47`，JPEG = `FF D8 FF`，GIF = `47 49 46 38`，WebP = `52 49 46 46 ... 57 45 42 50`
- WebP / GIF / TIFF 暂不支持，让用户传 image_path（plugin 报"unsupported image format, pass width/height/mime explicitly"）

**默认引入 dep**——`image-size` 是成熟纯 JS 库（DefinitelyTyped 维护），加它 weekly downloads 几百万。除非用户明确反对。

### 3.6 工具签名变化

```
send_image_as_user({
  chat_id: string,
  image_key?: string,           // 已有；现在变可选（带 image_path 时可省）
  image_path?: string,          // NEW：本地文件路径，自动 upload + 元数据
  width?: number,               // NEW：显式覆盖
  height?: number,              // NEW
  mime?: string,                // NEW
  root_id?: string              // 已有
})
```

校验：
- `image_key` 和 `image_path` 至少一个必给
- 给了 `image_path` 但没 `image_key` → 内部 upload 后填 key
- 三件套 (width/height/mime) 是 all-or-nothing：传了任一就要全传
- `size` (fileSize) **不在 MCP 工具表面暴露**——由 `image_path` 路径自动 stat，或缓存命中读出。如果只走 `image_key` + 显式 width/height/mime 路径而 fileSize 缺失：等抓包确认 fileSize 是不是必填。如果必填，则把它加进"all-or-nothing"里；如果可选，缺失时 proto 字段省略即可。**抓包阶段决定**（§7.6 同段记录）

### 3.7 `_sendMsg` IMAGE 支路改造

```js
// src/clients/user.js — 当前 IMAGE 错误路径（line 197）改造

async sendImage(chatId, imageKey, opts = {}) {
  // opts: { rootId, parentId, width, height, mime, size, thumbnailKey }
  const meta = {
    width: opts.width,
    height: opts.height,
    mimeType: opts.mime,
    fileSize: opts.size,
    thumbnailKey: opts.thumbnailKey,
  };
  return this._sendMsg(MsgType.IMAGE, chatId, { imageKey, imageMeta: meta }, opts);
}
```

工具层 `messaging-user.js::send_image_as_user` 负责按 §3.2 决议元数据，把元数据传进 sendImage。

### 3.8 现有 `send_image_as_user` 调用方升级路径

| 调用形式 | v1.3.8 行为 | v1.3.9 行为 |
|---|---|---|
| `(chat_id, image_key)`，key 是 v1.3.9 之后 upload 的 | HTTP 400 | ✅ 缓存 hit，自动元数据，发送成功 |
| `(chat_id, image_key)`，key 是 v1.3.8 升级前老 key | HTTP 400 | ❌ 缓存 miss，报错指向 §3.4 三个解决路径 |
| `(chat_id, image_path=...)` | 不支持参数 | ✅ 自动 upload + 元数据 + 发送 |
| `(chat_id, image_key, width, height, mime)` | 不支持参数 | ✅ 显式元数据，发送 |

老用户场景**没有静默失败**——要么 cache hit work，要么明确报错。

## 4. B.4 — `send_card_as_user` 真用户身份设计

### 4.1 Proto 变化（待捕获结果）

预测：

```proto
message Content {
  // 已有...
  optional bytes cardJson = ??;            // 字段号待抓包；可能是 string 而非 bytes
}
```

CARD 比 IMAGE 简单——卡片完全是 JSON，proto 层很可能就是个字符串 / bytes 字段装着 `JSON.stringify(card)`。但飞书也可能要附带额外字段（如 `cardId` / `templateId`），抓包确认。

### 4.2 工具签名（不变）

```
send_card_as_user({
  chat_id: string,
  card: object,                  // Feishu 卡片 JSON
  via?: "user" | "bot",          // 默认变 "user"（v1.3.8 默认 "bot"）
  root_id?: string
})
```

变化：
- `via` 默认从 `"bot"` 翻成 `"user"`——这是 v1.3.6 设计 fallback 时承诺过的
- `via="bot"` 保留作为显式 fallback（用于 user 路径出问题或者就是想 bot 身份发的场景）
- 工具描述里删除 `[v1.3.6+: bot-routed default]` 措辞；改为简单的 `[v1.3.9+: user-identity by default]`

### 4.3 `_sendMsg` CARD 支路（新增）

```js
// src/clients/user.js
async sendCard(chatId, card, opts = {}) {
  const cardBytes = Buffer.from(JSON.stringify(card), 'utf8');
  return this._sendMsg(MsgType.CARD, chatId, { cardJson: cardBytes }, opts);
}
```

### 4.4 默认 via 翻转的兼容性

v1.3.8 用户调 `send_card_as_user(chat_id, card)` 不带 via → 走 bot。
v1.3.9 同样调用 → 走 user（**不同身份显示**）。

这是**有意的行为变更**，但用户很可能希望以自己身份发卡片（这就是工具叫 `_as_user` 的原因）。Migration notes 里明确告知；不愿意切换的用户可以显式 `via="bot"`。

## 7. Open Questions / Risks

### 7.1 Capture 不可行
- IMAGE：极不可能失败——飞书 web 客户端每天发数百亿张图，一定能抓到
- CARD：i / ii / iii 三条路径，最差 iii 兜底；但 iii 可能要数天迭代
- **风险缓解**：B.1 / B.4 互相独立，单 ship 也能成立

### 7.2 飞书改协议
- 飞书 web 客户端版本升级可能改 proto 字段号 / 新增字段
- 我们没有"字段冻结"承诺；新版客户端 + 老 plugin 可能再次 HTTP 400
- **缓解**：`tests/fixtures/captured/` 保留 capture date，CHANGELOG 注明此风险存在

### 7.3 大图片元数据缓存膨胀
- LRU cap 1000 entries × ~200 bytes = ~200 KB，可接受
- 不需要持久化清理任务

### 7.4 image_key 跨 profile 共享
- v1.3.9 SSOT 下，profile A 上传的 image_key 在 profile B 是否可见？
- 答案：image_key 是租户级，profile 不同租户 → 不可见。缓存按 image_key 索引会 hit 错的元数据吗？不会，imageKey 在不同租户里命名空间是隔离的（前缀 img_v3_xxx 对每个租户独立生成）
- 但**理论上**两个租户可能撞 key（极小概率）。缓存里加 `tenant_id` 字段是过度防御——不做

### 7.5 image-size dep 安全性
- image-size 是 Image header parser，吃恶意输入可能 OOM / panic
- 缓解：plugin 调用前 `fs.statSync` 文件大小，> 50 MB 直接拒绝（图片应该没这么大）
- 不在 v1.3.9 spec 定上限，implementation 时加

### 7.6 thumbnail_key / fileSize 是否必填
- 如果飞书 IMAGE 元数据**必填** thumbnail_key，需要先生成缩略图再 upload——`sharp` lib 引入大 dep（250 MB native），不想要
- 多数 IM 协议允许省略 thumbnail_key，飞书很可能也是
- 同样的不确定性应用于 fileSize（§3.6 末尾）
- **抓包阶段验证**：用同一张图发多次但传不同子集字段，看哪些是必填、哪些可省。结果回填到 §3.6 校验逻辑

### 7.7 Live test 不在 CI
- 接受这个限制——飞书 cookie 不能放进 GitHub Secrets（一是会过期，二是一个人的身份发到自聊不能由 CI 触发）
- 替代：fixture-based regression（§5.1, §5.2）保住"wire format 没被回归"，live test 由发版者本地确认

# v1.3.7 Phase C — Bug Fixes + Tool Consolidation + New Write Tools Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Close out v1.3.6's testing-discovered bugs (14), consolidate the tool surface (delete 4, merge ~6 net), and add the long-deferred write tools for calendar (5) + Tasks (7) + wiki (5) + okr (3).

**Branches (5 sub-PRs, mostly parallel after C-bugs lands):**
- `fix/v1.3.6-bugs` (C1) — must land first; later sub-PRs build on the fixed state
- `refactor/tool-consolidation` (C2 + C3) — runs after C1

- `feat/wiki-okr-writes` (C5)
- `chore/test-residue-cleanup` (C6) — runs anytime
- `docs/test-methodology` (C8) — runs anytime

**Out of scope (handled in B):** any prompt/credentials/hooks work. C touches tool schemas + handlers + client methods only.

### Bug List + Fix Procedure

For each bug below: write a one-shot reproduction script (`scripts/repro-<bug-id>.js`), confirm reproduction, write the fix, run smoke, run the repro, commit.

#### C1.1: `send_image_as_user` — IMAGE protobuf encoding fails (status 0)

- [ ] **Repro:** `node -e "const c = require('./src/clients/user'); ..."` — call `sendMessage` with `msg_type=IMAGE`, observe status=0.
- [ ] **Investigation:** open `src/clients/user.js::_sendMsg`, find the `IMAGE` branch. Compare against the Feishu web client's actual protobuf payload (use Chrome DevTools network tab, copy the `messenger/cli_msg_send` POST body). Decode with `protobufjs`.
- [ ] **Likely cause:** image_key field number wrong, or the wrapping `RichText.elements[*].kind` discriminator is missing the IMAGE case.
- [ ] **Fix:** patch `src/clients/user.js::_sendMsg` IMAGE encoder.
- [ ] **Verify:** repro script now succeeds. Image lands in 飞书plugin测试群.

#### C1.2: `send_audio_as_user` — same pattern as IMAGE

- [ ] **Repro:** call `sendMessage(chat_id, audio_key, { msg_type: 'AUDIO' })`.
- [ ] **Fix:** patch `_sendMsg` AUDIO branch.
- [ ] **Note:** if too costly, fall through to **C2.2** (delete audio_as_user).

#### C1.3: `send_sticker_as_user` — any sticker_id rejected

- [ ] **Repro:** call with a known sticker_id (grab one from a real Feishu chat). Observe rejection.
- [ ] **Investigation:** sticker IDs may need a different protobuf field path or a separate API endpoint.
- [ ] **Decision:** likely cheaper to delete (C2.1). Document this in commit message before deleting.

#### C1.4: `send_as_user` / `send_post_as_user` reject `oc_xxx` chat IDs

- [ ] **Repro:** `await client.sendMessage('oc_xxxxxx', 'hi')` returns "invalid chat_id".
- [ ] **Cause:** the protobuf gateway expects numeric chat_id; oc_xxx is the open API format.
- [ ] **Fix:** in `src/clients/user.js::sendMessage` (and `sendPost`), detect `oc_xxx` prefix and call the new helper `_resolveOcToNumeric(ocId)` which calls `getOfficialClient().getChatInfo(ocId)` — the response contains the numeric `chat_id` field. Cache the mapping.
- [ ] **Alternative:** if no UAT/app available, fall back to `search` and find by chat name.

#### C1.5: `forward_message` — schema missing `receive_id_type`

- [ ] **Repro:** call `forward_message(message_id, 'ou_yyy')` — fails because default `receive_id_type=chat_id`.
- [ ] **Fix:** add to `src/tools/messaging-bot.js::schemas[forward_message]`:
  ```js
  receive_id_type: { type: 'string', enum: ['chat_id', 'open_id', 'union_id', 'user_id', 'email'], description: 'Type of receive_id (default: chat_id)' },
  ```
  Pass through in handler: `await ctx.getOfficialClient().forwardMessage(args.message_id, args.receive_id, args.receive_id_type)`.
- [ ] **Update method signature:** `clients/official/im.js::forwardMessage(messageId, receiverId, receiveIdType = 'chat_id')`.

#### C1.6: `forward_message` chokes on numeric IDs from `send_to_user`

- [ ] Same fix as C1.5 — auto-detect ID format if `receive_id_type` not provided. If numeric → use chat_id (P2P numeric is still chat_id from API perspective).

#### C1.7: `update_message` only works for `interactive`

- [ ] **Investigation:** Feishu API docs say `update_message` only supports `text` + `interactive`. Schema description says "text/post" — misleading.
- [ ] **Fix:** update schema description to reflect actual support. Reject other msg_types in handler with explicit error:
  ```js
  if (!['text', 'interactive'].includes(args.msg_type)) {
    return text(`update_message only supports msg_type=text or interactive (Feishu API limit). Got: ${args.msg_type}`);
  }
  ```

#### C1.8: `pin_message(pinned=false)` — wrong parameter location

- [ ] **Repro:** call with `pinned=false`, observe error.
- [ ] **Cause:** `clients/official/im.js::pinMessage` puts `message_id` in body, but unpin requires it in path: `client.im.message.unpin({ path: { message_id } })`.
- [ ] **Fix:** branch on `pinned`:
  ```js
  async pinMessage(messageId, pinned = true) {
    if (pinned) {
      return await this._safeSDKCall(() => this.client.im.message.pin({ data: { message_id: messageId } }));
    } else {
      return await this._safeSDKCall(() => this.client.im.message.unpin({ path: { message_id: messageId } }));
    }
  }
  ```

#### C1.9: `create_bitable_field` — UAT 1254001 / app 91403

- [ ] **Repro:** create a UAT-owned bitable, then `create_bitable_field(app_token, table_id, fieldConfig)`. Observe both UAT and app fail.
- [ ] **Investigation:** likely the `_uatREST` body serialization broke for nested objects. `git bisect` between v1.3.5 and v1.3.6 to find the offending commit.
- [ ] **Fix:** depends on bisect result. Most likely a JSON.stringify call that drops nested keys.

#### C1.10: `move_file` — 1061002 params error

- [ ] **Cause:** Feishu drive `move` requires `type` field (file/folder/docx/sheet/bitable/...).
- [ ] **Fix:** add `type` to schema (required), pass through:
  ```js
  // tools/drive.js move_file
  return text(`File moved: task=${(await ctx.getOfficialClient().moveFile(args.file_token, args.folder_token, args.type)).taskId}`);
  // clients/official/drive.js::moveFile(fileToken, folderToken, type)
  ```

#### C1.11: `delete_file` — 1062501 no permission

- [ ] **Cause:** uses `_safeSDKCall` (bot-only). UAT-owned files can't be deleted by bot.
- [ ] **Fix:** change `clients/official/drive.js::deleteFile` to use `_asUserOrApp`:
  ```js
  async deleteFile(fileToken, type) {
    return await this._asUserOrApp({
      uat: async (token) => this._uatREST('DELETE', `drive/v1/files/${fileToken}?type=${type || 'file'}`, null, token),
      app: async () => this._safeSDKCall(() => this.client.drive.file.delete({ path: { file_token: fileToken }, params: { type } })),
    });
  }
  ```

#### C1.12: `copy_file` — same pattern

- [ ] Change `_safeSDKCall` to `_asUserOrApp` so UAT-owned files can be copied.

#### C1.13: `manage_members(action=remove)` — 9499 object not supported

- [ ] **Repro:** call with `action=remove` and known group + open_ids.
- [ ] **Investigation:** maybe SDK version mismatch or `member_id_type` missing.
- [ ] **Fix:** add `member_id_type` to schema (default `open_id`). Pass through to client.

#### C1.14: `find_user` returns email/mobile only, no open_id

- [ ] Either fix or delete (C2.3). Recommended: delete since `search_contacts` covers the use case + returns more.

#### C1.15: `get_user_info` reports self as external tenant

- [ ] **Cause:** tenant_key comparison logic in `clients/official/contacts.js::getUserById` likely wrong, or the fallback hits cookie cache before the official API call completes.
- [ ] **Fix:** make the official API path always run first. Cache hit only if API returns 404/403. Ensure `tenant_key` comparison does string equality not object identity.

#### C1.16: `get_wiki_node` rejects `search_wiki` results

- [ ] **Cause:** `search_wiki` returns underlying `obj_token`; `get_wiki_node` only accepts `wiki_node_token` (different ID space).
- [ ] **Fix:** make `get_wiki_node` accept either:
  ```js
  // tools/wiki.js handlers.get_wiki_node
  // Try as wiki node token first; if "not found", try as obj_token via reverse lookup.
  ```
  Document in schema description: "Accepts either a wiki node token (from list_wiki_nodes) or an obj_token (from search_wiki) — handler tries both."

#### C1.17: `list_wiki_spaces` silently returns empty

- [ ] **Cause:** bot lacking `wiki:wiki:readonly` scope.
- [ ] **Fix:** detect empty + missing scope at the API level. Use the SDK error code (or proactively check scopes via app info endpoint at startup).
- [ ] **Better:** print a clear error: "App lacks `wiki:wiki:readonly`. Re-run `npx feishu-user-plugin oauth` to add the scope."
